# Design: 일감 토론 (인앱 채팅 + 외부 통화 룸 자동 생성)

Generated by /office-hours on 2026-05-15
Status: APPROVED
Mode: Builder
Approach: A (채팅 인앱 + 외부 룸 링크 자동 생성)

## Problem Statement

기획자·개발자가 일감 단위로 "회의급은 아닌 빠른 확인"을 할 때, 외부 도구(슬랙 허들 / Google Meet / Teams)로 이동하면 매번 일감 컨텍스트가 끊긴다. 일감 번호와 화면을 다시 공유하며 "어디 얘기 중이었지?"를 반복함.

## What Makes This Cool

- 채팅·통화 흔적이 **일감에 자동 결합** — 외부 도구로는 못 만드는 차별점
- "토론 시작" 한 번 클릭으로 같은 일감 안에서 텍스트 + 통화(외부 룸) 모두 켜짐
- 통화 종료 후에도 채팅 트레일이 일감에 그대로 남아 **다음 인계자에게 컨텍스트 자동 전달**
- 백그라운드에 있어도 진행 중 토론 시작 시 OS 알림 + 작업표시줄 깜빡임 (기존 알림 시스템 재사용)

## Constraints

- 백엔드: Spring Boot 3.2 + JPA, SSE 운영 중 (`SseEmitterRegistry`)
- 프론트엔드: React 18 + Vite, `react-markdown` + `MentionInput` 재사용 가능
- DB: H2(dev) / PostgreSQL(prod), 권한·멤버십 모델 기존 활용 (`project_members`)
- 참여 패턴: **대부분 1:1, 가끔 3명** (이 수치가 WebRTC 자체구현 보류의 근거)
- 외부 SaaS 사용 가능

## Premises (agreed)

1. 채팅 흔적은 일감에 영구 보존되어야 한다 (차별점) — 동의됨
2. 통화 자체는 검증된 외부 도구(Jitsi) 활용 — 동의됨
3. 인앱 풀 WebRTC는 가치 검증 전 ocean에 가까움 — 2단계로 미룸. 동의됨

## Approaches Considered

### Approach A: 채팅 인앱 + 외부 룸 링크 자동 생성 (선택)

- 채팅은 SSE로 실시간 동기화, 통화는 Jitsi 임시 룸 URL 자동 생성 → 첫 메시지로 첨부
- Effort: **S** (human ~3일 / CC ~1-2시간)
- Risk: 낮음
- 비용: $0 (Jitsi 공개 인스턴스 `meet.jit.si` 무료, 가입 없이 룸 생성)

### Approach B: 풀 WebRTC P2P (채팅 + 음성 + 화면공유 인앱)

- SSE 시그널링 + Google STUN, P2P mesh
- Effort: **L** (human ~2주 / CC ~6-8시간)
- Risk: 중 (NAT 통과 실패 / 브라우저 호환 / 권한 UX)

### Approach C: 단계적 (A로 시작 → 사용량 보고 B로 확장)

- 1단계 A, 사용 데이터 누적 후 2단계 B
- Effort: 1단계 S, 2단계 L

## Recommended Approach: A

C도 합리적이지만 사용자 의사: "1단계만 본 세션에서 끝낸다." 2단계(WebRTC)는 추후 별도 설계.

근거: Jitsi 임시 룸은 사실상 완성품이고, 통화 인앱 매립의 한계 가치(컨텍스트 유지)는 이미 채팅으로 90% 달성된다. WebRTC의 엣지케이스(기업망 NAT 통과 실패, Safari getDisplayMedia 차이, 마이크/화면공유 권한 UX 데드락)는 데이터 없이 베팅할 만한 가치가 부족.

## 구현 설계

### 데이터 모델

**기존 `comments` 테이블 확장** (별도 chat_messages 만들지 않음 — 단순성):

```sql
ALTER TABLE comments ADD COLUMN IF NOT EXISTS type varchar(20) NOT NULL DEFAULT 'COMMENT';
ALTER TABLE comments ADD COLUMN IF NOT EXISTS discussion_id bigint;
ALTER TABLE comments ADD CONSTRAINT comments_type_check
  CHECK (type IN ('COMMENT','CHAT','SYSTEM'));
```

- `type = 'COMMENT'`: 기존 비동기 댓글 (변경 없음)
- `type = 'CHAT'`: 토론 채팅 메시지 (discussion_id로 묶임)
- `type = 'SYSTEM'`: 토론 시작/종료, Jitsi 링크 자동 생성 메시지

**신규 `discussions` 테이블**:

```sql
CREATE TABLE discussions (
  id          bigserial PRIMARY KEY,
  todo_id     bigint NOT NULL REFERENCES todos(id),
  started_by  bigint NOT NULL REFERENCES users(id),
  started_at  timestamp(6) NOT NULL,
  ended_at    timestamp(6),
  room_url    varchar(500),
  CONSTRAINT uk_discussions_todo_active UNIQUE (todo_id, ended_at)
  -- ended_at IS NULL인 행이 todo당 1개 — H2/Postgres 부분 unique 패턴 검토
);
CREATE INDEX idx_discussions_todo ON discussions(todo_id, started_at DESC);
```

운영 적용 시 PostgreSQL에서는 unique partial index가 더 적절:
```sql
CREATE UNIQUE INDEX uk_discussions_active_per_todo
  ON discussions(todo_id) WHERE ended_at IS NULL;
```

### API 엔드포인트

| 메서드 | 경로 | 동작 |
|---|---|---|
| POST | `/api/todos/{id}/discussions` | 토론 시작 — Discussion 생성, Jitsi 룸 URL 생성, SYSTEM 메시지로 링크 첨부, 담당자/작성자 알림 |
| POST | `/api/discussions/{id}/messages` | CHAT 메시지 추가 → SSE push |
| POST | `/api/discussions/{id}/end` | 토론 종료 — `ended_at` 기록 |
| GET | `/api/todos/{id}/discussions` | 토론 히스토리 (최신순) |
| GET | `/api/discussions/{id}/messages` | 특정 토론의 메시지 목록 |

### Jitsi 룸 URL 생성 규칙

```
https://meet.jit.si/Kanban-Todo-{todoId}-{shortUuid}
```

- `shortUuid`: 토론 생성 시 8자 랜덤 (충돌·URL 추측 방지)
- 룸 이름에 `todoId` 포함 — Jitsi 측 분석/네트워크 디버깅 시 식별 용이
- 별도 인증 없음 (Jitsi 공개 인스턴스 정책 그대로) — 룸 이름 추측 어려움 + 일감 권한자만 URL 알 수 있음

### SSE 이벤트

| 이벤트 | 페이로드 | 수신 대상 |
|---|---|---|
| `discussion_started` | `{ todoId, discussionId, startedBy, roomUrl }` | 일감 담당자 + 작성자 + 프로젝트 멤버 |
| `chat_message` | `{ discussionId, messageId, authorId, content, createdAt }` | 해당 토론 참여자 (프론트가 discussion_id 구독) |
| `discussion_ended` | `{ todoId, discussionId, endedAt }` | 동일 |

### 알림 (NotificationService 확장)

- 새 NotificationType: `DISCUSSION_STARTED`
- 메시지: `[프로젝트] {actor}님이 [{todo.summary}] 토론을 시작했습니다 — 클릭하여 참여`
- 본인 액션 제외 로직 유지 (기존 패턴)
- 권한 허용 상태면 Windows OS 알림 + 작업표시줄 깜빡임 자동 (`useNotifications` 기존 통합)

### UI/UX

**일감 상세 우측 상단**:
- 진행 중 토론 없을 때: `[토론 시작]` 버튼
- 진행 중 토론 있을 때: `🔴 토론 중 (N명 참여) — 참여하기` 배지

**토론 패널** (`DiscussionPanel` 신규 컴포넌트):
- 상단: Jitsi 룸 링크 버튼 (새 창 열기) + "토론 종료" 버튼 (시작자 또는 프로젝트 마스터만)
- 가운데: 채팅 스레드 (메시지별 아바타·이름·시각·마크다운 렌더링, MentionInput 재사용)
- 하단: 메시지 입력창 (Enter 전송, Shift+Enter 줄바꿈, @멘션 지원)
- SYSTEM 메시지는 옅은 회색 + 아이콘으로 구분 ("토론 시작됨 — Jitsi 룸 생성", "토론 종료됨")

**토론 종료 후**:
- 패널은 그대로 보임 (히스토리)
- "이 토론은 종료되었습니다" 라벨 + 입력창 비활성화
- 같은 일감에서 다시 `[토론 시작]` 가능 → 새 Discussion 행 생성

### 권한 모델

| 동작 | 권한 |
|---|---|
| 토론 시작 | 프로젝트 멤버 |
| 메시지 작성 | 프로젝트 멤버 (토론이 진행 중일 때만) |
| 토론 종료 | 시작자 OR 프로젝트 마스터 |
| 토론 히스토리 조회 | 프로젝트 멤버 |

## Open Questions

- 메시지 편집/삭제 정책 → **추천**: 댓글과 동일 (작성자만 삭제, 편집 불가)
- 채팅 메시지 무한 보관 → 스토리지 부담 거의 없음, 그대로 유지
- 화면공유 캡쳐 보존 → Jitsi는 클라이언트 녹화 가능, 자동 저장은 별도 인프라 필요. 1단계 범위 외
- 통화 종료 후 Jitsi 룸 URL을 비활성화? → Jitsi 측 만료 없음. UI에서 종료된 토론의 룸은 흐리게 표시 + "이 룸은 종료됨" 안내 (재입장은 기술적으로 가능하나 권장 안 함)
- 모바일 UX → 일감 상세가 모바일에서 어떻게 보이는지 별도 확인 필요

## Success Criteria

- [`토론 시작` 클릭 → Jitsi 룸 입장까지 < 10초]
- [동일 일감의 토론 종료 후 채팅 히스토리 100% 보존]
- [일감 상세 페이지 재진입 시 진행 중 토론에 자동 합류 가능]
- [백그라운드 상태의 일감 담당자가 토론 시작 알림을 누락 없이 수신]

## Next Steps (구현 순서)

2. **백엔드 엔티티** — `Discussion`, `Comment` 엔티티 확장, 리포지토리, 서비스
3. **백엔드 컨트롤러** — 위 API 5개 엔드포인트, Jitsi URL 생성 유틸
4. **백엔드 SSE 확장** — `chat_message`, `discussion_started`, `discussion_ended` 이벤트
5. **백엔드 알림 확장** — `DISCUSSION_STARTED` NotificationType, NotificationService 메서드
6. **프론트엔드 API** — `frontend/src/api/discussions.js`
7. **프론트엔드 컴포넌트** — `DiscussionPanel` (채팅 UI), 일감 상세에 통합
8. **프론트엔드 SSE 핸들러** — `chat_message` 수신 → 채팅 갱신, `discussion_started` → 배지 갱신 + 알림
9. **검증** — 두 계정 동시 접속 → 토론 시작 → Jitsi 연결 → 채팅 동기화 → 종료 → 히스토리 보존

## What I noticed about how you think

- 첫 메시지가 "가능할까?"였다는 게 인상적 — 대부분 "이거 만들어줘"로 시작하는데, 가능성과 트레이드오프를 먼저 점검하는 습관이 보임
- "회의까지는 좀 애매하고" — 이미 use case의 정확한 무게감을 잡고 있음. 회의도 댓글도 아닌 빈 슬롯을 본 시선
- 프리미스 도전(외부 도구 안 쓰는 이유?)에 즉시 본질적인 답: "컨텍스트가 매번 끊김". 추상적 불만이 아니라 구체적 마찰 지점
- 풀 WebRTC 안 가고 "1단계만 끝내자" 결정 — 검증 안 된 인프라에 큰 베팅 안 한 절제. "끝(End)"이라고 못 박은 게 핵심
